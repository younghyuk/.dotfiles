# 기본
- 한국어로 답변
- 큰 변경 전에 접근법 먼저 설명하고 확인받기

# 코드 스타일
- 주석은 이유가 non-obvious할 때만 (what이 아닌 why)
- 에러 핸들링은 시스템 경계(사용자 입력, 외부 API)에서만
- 실제로 필요할 때만 추상화
- 현재 요청 범위 밖의 리팩터링, 기능 추가 금지

# 시크릿 관리
- 개인 시크릿: 1Password (`op://` 참조, `op run --` 래퍼)
- 회사 시크릿: Infisical (`infisical run --` 래퍼)
- 환경 변수나 MCP 서버 설정 시 위 도구에 맞게 구성

# 작업 방식
- DB 마이그레이션, 스키마 변경 등 코드 외 되돌리기 어려운 작업 전에 확인
- 테스트/타입체크는 코드 변경 후 실행

# PR 생성
- 제목: 변경의 핵심을 한 줄로 (70자 이내)
- 본문은 what보다 why 중심으로 작성
  - 왜 이 변경이 필요한가 (배경, 문제 상황)
  - 어떤 결정을 했고 왜 그 방향인가 (대안 대비 이유)
  - 리뷰어가 집중해야 할 부분
- 관련 이슈/티켓 번호 명시
- 스크린샷·로그는 UI 변경이나 동작 확인이 필요한 경우에만 첨부
